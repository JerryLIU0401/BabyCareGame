# 企劃文件與目前腳本邏輯比對

本文件比對 `Assets/Reference/《哈姆的健康生活》說明書AR版.pdf` 與目前 `Assets/Scripts/` 的實作狀態。目標是讓後續開發能清楚知道哪些功能已存在、哪些只是舊版或臨時實作、哪些需要依新企劃補齊。

## 企劃來源

- 文件：`Assets/Reference/《哈姆的健康生活》說明書AR版.pdf`
- 頁數：3 頁
- 主題：AR 桌遊玩法說明
- 遊戲核心：玩家透過情境卡、步驟卡、功能卡與命運卡進行回合制遊戲，完成情境後使用平板 APP 掃描卡牌播放 AR 動畫並記錄分數。

## 舊資產定位

`Assets/Old/` 內包含舊版 ImageLibrary、卡牌圖片、UI 圖片與相關 `.meta` 檔。這些檔案不代表新企劃需求，只能作為新版 `ImageLibrary` 的 Unity 資產格式、分組方式與命名方式參考。

| 項目 | 可否作為需求依據 | 說明 |
| --- | --- | --- |
| `Assets/Old/ImageLibrary` | 否 | 僅可參考 `XRReferenceImageLibrary` 的資產格式與目錄結構。 |
| `Assets/Old/Image` | 否 | 屬於舊版圖片資產，不應直接視為新 UI 或新卡牌需求。 |
| `Assets/Old` 內 `.meta` | 否 | 僅維持舊資產 GUID，不應搬移或覆蓋到新版資產。 |

## 目前程式核心

目前腳本比較接近「選擇玩家 -> 掃描 AR 卡牌 -> 依模型分數加分 -> 結算」的流程。這個流程已具備玩家資料、UI 更新、AR 圖像追蹤、Prefab 生成與分數更新，但尚未完整支援企劃中的桌遊卡牌系統。

## 需優先補齊的模組

| 優先度 | 建議模組 | 原因 |
| --- | --- | --- |
| 高 | 卡牌資料模型 | 企劃中的情境卡、步驟卡、功能卡、命運卡都需要明確資料結構。 |
| 高 | 牌庫與手牌管理 | 抽牌、補牌、棄牌與 PASS 都依賴手牌與中央牌庫。 |
| 高 | 勝利條件服務 | 目前固定 30 分，需改為依玩家人數與時間/任務狀態判定。 |
| 中 | 情境進度管理 | 需要記錄場上 4 張情境卡、步驟填入進度與完成狀態。 |
| 中 | AR 掃描結果對應 | 需從單一模型加分改為掃描已完成情境與步驟組合後播放動畫。 |
| 中 | 遊戲倒數計時器 | 企劃明確要求 20 分鐘時間限制。 |

## 後續開發注意事項

- 不建議直接在 `GameManager` 補滿全部卡牌規則，應拆分為牌庫、手牌、情境進度與勝利判定等單一職責模組。
- `PlayerData.scanCount` 目前存在但使用較少，若要承接企劃的掃描次數或回合限制，需要先確認規則語意。
- `ModelInfo` 目前可作為 AR 掃描物件資料來源，但新企劃可能需要更細的卡牌型別與情境對應資料。
- `Assets/Old/ImageLibrary` 只適合用來確認 Unity ImageLibrary 資產格式，不應直接導入舊版卡牌內容。
